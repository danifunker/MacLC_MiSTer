# Mac LC core: V8/CPU/Audio fidelity pass

## Context

The V8 schematic audit (124 pins from `schematics/LC_Master_Netlist.csv`) revealed several gaps between the real Macintosh LC hardware and the current MiSTer core. The CPU core (TG68K) is correctly configured for the 68020 instruction set (`status_cpu = 2'b10` at `MacLC.sv:224`), but the **bus wrapper** in `rtl/tg68k/TG68K.v` is hard-wired for a 68000-style bus and discards the upper address bits, FC3, SIZ, and DSACK. Several other V8 functions are stubbed or run on legacy Mac Plus code paths (notably audio). The most damaging consequence is that **bus errors on unmapped accesses are never raised**, which is the prime suspect for the FPGA boot stalling in `0xCxxxxx` per the `fpga-boot-debug` memory.

This plan addresses, in order of impact:

1. CPU bus path (widen address, expose 68020 bus signals where TG68 actually has them)
2. BERR on unmapped accesses
3. Audio path (route ASC samples to `AUDIO_L`/`AUDIO_R`, retire the Mac Plus ring buffer)
4. CULTDAC0 / DFAC_CLK / SND[0:2] — document or implement
5. **SCC clocking via V8 SCSI_PCLK** — precondition for AppleTalk over LocalTalk; the LC has no SCC crystal, the SCC's `RTxC` clock is generated by V8 pin 124 and our `scc.v` doesn't model it at all
6. **Simulator parity** — every change above must be mirrored in `verilator/sim.v`, which is a parallel top-level (not a wrapper). This step is folded into each commit, not done separately.
7. Loose ends: `slot_irq`, monitor sense, schematic label fix

The plan deliberately does **not** try to make the bus pin-faithful with full DSACK/SIZ — TG68's wrapper presents a 68000-style cycle and the V8 emulation in `addrController_top.v` already handles dynamic sizing implicitly via `_memoryUDS`/`_memoryLDS`. The goal is correctness, not pin-level mimicry.

## Step 1 — Widen CPU address bus to 32 bits

**Files:** `MacLC.sv`, `rtl/addrController_top.v`, `rtl/dataController_top.sv`, `rtl/addrDecoder.v`

**Why:** TG68 already produces `tg68_a[31:0]` (`TG68K.v:35`), but `MacLC.sv:702` declares `wire [23:0] cpuAddr` and `MacLC.sv:778` only wires `cpuAddr[23:1] = tg68_a[23:1]`. Bits 31:24 are silently dropped, which means stray high-address fetches (a 68020 in 24-bit mode still emits the full address) go unnoticed. Catching them in the trace log is a prerequisite for diagnosing the `0xCxxxxx` stall.

**Changes:**
- `MacLC.sv:702`: change to `wire [31:0] cpuAddr;`
- `MacLC.sv:778`: `assign cpuAddr[31:1] = tg68_a[31:1];`
- `MacLC.sv:763–764`: keep the 24-bit decode (`cpuAddr[23:21]==3'b111`) but also force BERR (next step) when `|cpuAddr[31:24]`.
- `addrController_top.v:15`: widen `input [23:0] cpuAddr` → `[31:0] cpuAddr`. Internally, classification (`motherboard_high`, `in_simm`, etc.) keeps using `cpuAddr[23:0]` so no functional change for in-range accesses.
- `addrDecoder.v`: same widening of the `address` port; pass `address[31:24]` through to a new `selectUnmapped` term that fires when any high bit is set.
- Add a `$display` in `VERBOSE_TRACE` mode whenever `|cpuAddr[31:24]` so the trace log surfaces high-addr accesses.

**Risk:** Low. All existing slices use `[23:0]`, so widening is additive.

## Step 2 — Real BERR on unmapped accesses

**Files:** `MacLC.sv`, `rtl/addrController_top.v`, `rtl/addrDecoder.v`, `rtl/tg68k/TG68K.v` (read-only verify)

**Why:** TG68K already takes a `berr` input (`TG68K.v:32`, passed straight to `TG68KdotC_Kernel.berr` at line 216). Today `MacLC.sv:821` ties it to `cpuFC == 3'b111` (autovector hack only). A real V8 asserts BERR for any unmapped CPU access; the absence of this is almost certainly why the 68020 wedges in `0xCxxxxx`.

**Changes:**
- `addrDecoder.v`: confirm `selectUnmapped` already covers the cases we need (it does — `selectUnmapped` is generated alongside the other `select*` lines and is plumbed up to `addrController_top.v:46`). Extend it so any `cpuAddr[31:24] != 0` also asserts unmapped.
- `MacLC.sv:821`: replace
  ```
  .berr ( cpuFC == 3'b111 )
  ```
  with a new wire:
  ```
  wire cpu_berr = (cpuFC == 3'b111)                       // keep autovector path
                 || (!_cpuAS && cpuBusControl && selectUnmapped);
  .berr (cpu_berr)
  ```
  Gate on `cpuBusControl` so the BERR fires inside the CPU bus slot, matching how `_cpuDTACK` is gated at `MacLC.sv:764`.
- Add a `$display` (under `SIMULATION`) when BERR fires for unmapped, capturing `cpuAddr`, `cpuFC`, `_cpuRW`. This is the diagnostic that the `fpga-boot-debug` work was missing.

**Verification:** Run Verilator with the existing `check_boot.sh --run 100`. Expect either:
- The trace log now shows a BERR exception followed by a vector fetch from `$00008` (bus error vector), OR
- Boot proceeds further than before because the CPU was previously hanging on a read that should have errored.

**Risk:** Medium. If we BERR on something the ROM legitimately polls, boot will regress. Mitigation: gate the new BERR on `cpuBusControl && !_cpuAS` so only completed bus cycles trigger it, and add a `$display` so any spurious fire is visible in `cpu_trace.log`.

## Step 3 — Replace legacy ring-buffer DMA with ASC sample output

**Files:** `rtl/asc.sv`, `rtl/dataController_top.sv`, `rtl/addrController_top.v`, `MacLC.sv`

**Why:** Per the Phase 1 audit, `rtl/asc.sv` is a register-only stub: it implements VERSION/MODE/CONTROL/FIFO_STAT for ROM probing but produces no PCM output. `dataController_top.sv:125–147` still drives `audioOut` from the Mac-Plus-style 8-bit DMA latch reading `memoryDataIn[15:8]`, fed by the `audioAddr` ring buffer at SDRAM `$07FE80` (`addrController_top.v:71–99`). On a real LC, ASC owns the entire audio path; the Mac Plus sound buffer doesn't exist.

**Changes (incremental):**

**3a. Extend `rtl/asc.sv` with an actual FIFO + sample output.**
- Add two 1024-byte FIFOs (A and B channel) — real ASC has 2× 1 KB FIFOs at `$F14000–$F147FF` and `$F14800–$F14FFF`. CPU writes go through `cs && we && addr[11]==0` paths. The existing `addr[11:0]` decode at `asc.sv:20–26` lines up.
- Add a sample-rate clock enable (real ASC runs ~22.255 kHz; we can derive it from `clk_sys` with a divider, or reuse `clk8_en_p` and divide further).
- Add ports: `output reg signed [15:0] sample_l`, `output reg signed [15:0] sample_r`, `output sample_tick`. On each tick, pop one byte from each FIFO (or one stereo sample), sign-convert from unsigned 8-bit, and present.
- Update the FIFO_STAT register to reflect real fill levels so the ROM's playback test passes.
- Keep the existing `irq` output: assert when FIFO half-empty (matches Sansa behavior closely enough for the LC ROM's purposes).

**3b. Wire ASC samples into the top-level audio mux.**
- `MacLC.sv:969–978`: add the new outputs.
- `MacLC.sv:650–651`: change
  ```
  assign AUDIO_L = {audio[10:0], 5'b00000};
  assign AUDIO_R = {audio[10:0], 5'b00000};
  ```
  to
  ```
  assign AUDIO_L = asc_sample_l;
  assign AUDIO_R = asc_sample_r;
  ```
  Drop the `audio` wire entirely.

**3c. Retire the legacy DMA path.**
- `dataController_top.sv:125–147`: delete the `audio_x1`/`audio_x2`/`audio_x4`/`audio_latch`/`audioOut` block. Drop the `audioOut` port from the module and the corresponding wiring at `MacLC.sv:1052`.
- `addrController_top.v:71–99`: delete `audioAddr`, `snd_div`, `vblankD`, `vblankD2`, the `localparam SIZE/STEP`, and the `assign loadSound = sndReadAck` wire. Delete `sndReadAck` from the extra-slot mux at line 222 — slot 2 becomes free for future use (e.g., a second floppy, or just left idle).
- Drop the `loadSound` port from `addrController_top` and `dataController_top`. Dead-port at `MacLC.sv:742, 868, 1053`.

**Verification:**
- Verilator builds clean (no multi-driver errors).
- `make && ./obj_dir/Vemu --screenshot 350 --stop-at-frame 351 2>/dev/null 1>/dev/null` — frame 350 still shows the grey/black memory-test pattern (per `feedback_via_sr_simulation` memory).
- Boot to ROM startup chime: the chime is the ROM playing a known sample through ASC. If `asc_sample_l/r` toggles during boot the sim audio will move; `verilator/check_boot.sh` doesn't validate audio but the trace log should show writes to `$F14xxx`.

**Risk:** Medium-high. The ASC FIFO behavior is the part of the LC ROM that historically trips up emulators. Mitigation: keep step 3a behind a `define USE_ASC_AUDIO` so it can be reverted to the stub instantly while debugging.

## Step 4 — CULTDAC0 / DFAC_CLK / SND[0:2]

**Files:** `rtl/asc.sv` (comment), `rtl/ariel_ramdac.sv` (comment), `MacLC.sv` (comment)

**Why:** These three pin groups are V8↔DFAC and V8↔CULTDAC physical signals on the schematic. They have no semantic meaning above the chip boundary — DFAC is the analog audio chip the ASC drives through SND[0:2], and CULTDAC0 is the RAMDAC pixel clock fed from V8. Once Step 3 routes ASC samples directly to `AUDIO_L`/`AUDIO_R`, the SND[0:2] bus has no analog target to model. Same for DFAC_CLK. CULTDAC0 is replaced by `clk_sys` driving `ariel_ramdac` directly.

**Changes:** Documentation only.
- Add a comment block at the top of `rtl/asc.sv`: "Real ASC drives DFAC via SND0..2 + DFAC_CLK. In this core the ASC FIFOs are sample-rate-converted in-block and routed straight to AUDIO_L/R; no DFAC modelling is needed."
- Add a comment in `rtl/ariel_ramdac.sv`: "Real RAMDAC is clocked by V8's CULTDAC0 (pixel clock). Here we use clk_sys directly; the pixel-clock divider lives in maclc_v8_video.sv."
- Add a `// V8 schematic SND[0:2]/DFAC_CLK/CULTDAC0: see asc.sv / ariel_ramdac.sv` comment near the `asc_inst` block in `MacLC.sv:969`.

**Risk:** None.

## Step 5 — SCC clocking via V8 SCSI_PCLK (AppleTalk precondition)

**Files:** `rtl/scc.v`, `rtl/addrController_top.v` (or a new `rtl/v8_clocks.sv`), `MacLC.sv`, `rtl/dataController_top.sv` (SCC instantiation)

**Why:** The schematic shows the LC has **no separate SCC chip** — `UE9` is a Zilog **85C80** (combined NCR5380 SCSI + Z85C30 ESCC). The SCC half's three clock inputs (`/RTxCA` pin 15, `PLCK` pin 22, `/RTxCB` pin 29) are **all tied to the single net `SCSI_PCLK`**, which originates from **V8 pin 124 (`SCSI_RTXC`)**. There is no external crystal on the SCC. Our `rtl/scc.v` (line 20) only takes `input clk` (system clock) and never models RTxC, which is why AppleTalk-over-LocalTalk does not work: LocalTalk is 230.4 kbit/s SDLC with FM-0 encoding, and SDLC clocks come from RTxC (or the DPLL locked to RTxC), not from the BRG. Async UART works in our core because the BRG path happens to function with `clk` as its reference; SDLC has no such fallback.

`SCSI_PCLK` is documented (Apple LC tech notes + Egret schematics) to be **3.672 MHz** — exactly 16× the 230.4 kHz LocalTalk bit rate, derived in V8 from the `25.1750 MHz` oscillator on pin 42.

**Changes (this is an enabler — full AppleTalk needs Step 5d on top):**

**5a. Generate `scsi_pclk_en` in the V8 emulation layer.**
- Add a new tiny module `rtl/v8_clocks.sv` (or inline in `addrController_top.v`) that produces a 1-cycle-wide enable pulse `scsi_pclk_en` at 3.672 MHz from `clk_sys` (32.5 MHz). Use a fractional divider: 32.5e6 / 3.672e6 ≈ 8.851. A simple `+5` / `−4` accumulator (Bresenham) gives <1% error, which is well within SCC tolerances.
- Expose `scsi_pclk_en` as a top-level wire alongside `clk8_en_p`/`clk8_en_n`.

**5b. Refactor `scc.v` to use `rtxc_en` instead of `clk` for baud generation.**
- Add `input rtxc_en` to the `scc` port list. Wire it to `scsi_pclk_en` from MacLC.sv.
- Replace the `baud_divid_speed_a/b` constants (currently `24'd3385` for "9600 baud at 32.5 MHz") with values referenced against the **3.672 MHz** RTxC. The Z8530 async formula `BAUD = SourceClock / (2 × (BRG + 2) × ClockMode)` then produces correct rates without further fudging.
- All BRG counters tick on `clk_sys` only when `rtxc_en` is high. Existing async UART receivers/transmitters retain their current sub-bit timing but their *bit* timing is now driven by the divided rtxc rate. Verify the existing serial loopback test (`baud_divid_speed_a <= 24'd100` at line 1373) still passes.
- The `wr11_*` register (clock source select) becomes meaningful: if WR11 says "RxClk from BRG output," use the BRG; if it says "RxClk from RTxC," tick once per `rtxc_en`; if "RxClk from DPLL," route through Step 5d.

**5c. Update SCC instantiation.**
- `dataController_top.sv` (where `scc` is instantiated): add the new `.rtxc_en(scsi_pclk_en)` port.
- `MacLC.sv`: thread `scsi_pclk_en` from `addrController_top` (or `v8_clocks` instance) through `dataController_top`.

**5d. SDLC + DPLL support in `scc.v` (the hard part — separate commit).**
- This is the actual AppleTalk enabler. Implement:
  - **WR4 SDLC mode** decode (bits 5:4 = 2'b10 → SDLC).
  - **FM-0 encode/decode** (Manchester-like, 1=transition mid-bit, 0=transition only at edge).
  - **Flag (`0x7E`) detection** + zero-bit insertion/deletion.
  - **CRC-16** (CCITT polynomial `0x1021`).
  - **DPLL** locked to incoming FM-0 transitions for receive clock recovery (the DPLL itself is a counter that resyncs on every transition; reasonably small RTL).
- This is genuinely new code; the existing `rtl/uart/` modules are async-only and can't be reused for SDLC. Reference: Z8530 datasheet section "SDLC Operation" and Apple's `LocalTalk Driver` source from MPW.

**Risk:** 5a–5c are medium risk — touching SCC clocking can break the existing async serial path (which is the only thing that currently works for the boot console). Mitigation: keep the old `clk`-based BRG path alive behind `define SCC_USE_RTXC` until 5b is verified to not regress async serial. 5d is high risk and high effort — it should be its own multi-commit effort, scheduled after 5a–5c are stable.

**Verification:**
- After 5a–5c: serial console output during boot must still appear (existing async path).
- `verilator/check_boot.sh --run 100` must still PASS.
- After 5d: AppleTalk LAP test from MacOS Network control panel; expect the LC to claim a node ID and respond to AEP echo.

## Step 6 — Simulator parity (`verilator/sim.v`)

**Files:** `verilator/sim.v`, `verilator/sim_main.cpp` (read-only verify)

**Why:** `verilator/sim.v` is a **parallel top-level** that mirrors `MacLC.sv` for the Verilator harness. It is **not** a thin wrapper around `MacLC.sv` — it duplicates the wiring (it instantiates `tg68k`, `addrController_top`, `pseudovia`, `maclc_v8_video`, `asc`, `dataController_top`, `sdram` directly). Every change we make to `MacLC.sv` in Steps 1–5 must be replicated here or the simulator will diverge from the FPGA build, which defeats the entire reason we run Verilator boot tests.

Confirmed today by inspection of `verilator/sim.v`:
- Line 213: `wire [23:0] cpuAddr;` — same 24-bit truncation as `MacLC.sv`
- Line 270: `assign cpuAddr[23:1] = tg68_a[23:1];` — same dropped high bits
- Line 315: `.berr ( 1'b0 )` — **even worse than MacLC.sv** (which at least uses `cpuFC==3'b111`); the Verilator build never raises bus errors at all
- Line 176–178: same legacy `wire [10:0] audio` / `AUDIO_L = {audio,5'b0}` / `AUDIO_R = ...`
- Line 234: `wire loadSound;` and 426: `.loadSound(loadSound)`
- Line 484–492: `asc_inst` with same register-only port set
- Line 571–572: `.audioOut(audio)` / `.loadSound(loadSound)` — legacy DMA path
- Line 752: `assign debug_cpuAddr = cpuAddr;` exposed as 24-bit to `sim_main.cpp`

`sim_main.cpp` consumes:
- `debug_cpuAddr` as `uint32_t` (line 270, 306) — already wide enough, no C++ change needed when we widen
- `AUDIO_L` / `AUDIO_R` via `audio.Clock(...)` (line 341) — port name unchanged across all steps, so no C++ change needed for the audio rewiring

## Verification plan (end-to-end)

1. **Quartus-clean check.** After steps 1+2, grep for any new multi-driver `always` blocks (per CLAUDE.md). The widening is additive so this should be clean, but verify.
2. **Verilator boot smoke test.**
   ```
   cd verilator && make clean && make
   ./obj_dir/Vemu --screenshot 350 --stop-at-frame 351 2>/dev/null 1>/dev/null
   ```
   Frame 350 must still show the grey/black memory-test pattern (per `feedback_via_sr_simulation`). If it goes uniform grey, the BERR change broke Egret comms — back out step 2 first.
3. **`check_boot.sh --run 100`** — must still PASS.
4. **High-address trace check.** `grep "HIGH_ADDR" verilator/cpu_trace.log` (the new `$display` from step 1). Should be empty during normal boot; if anything fires, that's a real bug we caught.
5. **BERR diagnostic.** `grep "BERR_UNMAPPED" verilator/cpu_trace.log`. Expected: occasional fires during ROM probing of nonexistent slots, followed by a clean exception return. If BERR fires storm, gate it more tightly.
6. **Audio sanity.** `grep "F14" verilator/cpu_trace.log` — confirm the LC ROM is writing to ASC FIFO addresses during boot. After step 3, `asc_sample_l/r` should have non-zero activity by the chime point.
7. **FPGA boot regression.** Once Verilator passes, build Quartus and check whether the FPGA still wedges in `0xCxxxxx`. If step 2 worked, the wedge should now be a clean BERR exception (visible in the trace overlay) instead of a silent hang.

## Order of execution & checkpoints

I recommend committing after each step so we can bisect if anything regresses:

1. **Commit A:** Steps 1 + 2 (widen address bus + real BERR). Validate with checkpoints 2, 3, 4, 5, 7. **This is the highest-value change** — it's the one most likely to unstick the FPGA boot.
2. **Commit B:** Step 3a (extend ASC with FIFO + sample output, no top-level rewiring yet). Validate ASC standalone via the trace log (`grep F14`).
3. **Commit C:** Steps 3b + 3c (rewire AUDIO_L/R, retire legacy DMA). Validate checkpoints 2, 3, 6.
4. **Commit D:** Step 5a + 5b + 5c (V8 SCSI_PCLK generator, SCC `rtxc_en` refactor). Async serial must still work.
5. **Commit E:** Step 5d (SDLC + DPLL in scc.v). This is the AppleTalk-enabling commit and is expected to take multiple sub-commits.
6. **Commit F:** Steps 4 + 7 (documentation + cosmetic). Validate checkpoint 2.

**Every commit above must touch both `MacLC.sv` AND `verilator/sim.v` in lockstep.** The Step 6 table is the master cross-reference. Verilator boot smoke test (`./obj_dir/Vemu --screenshot 350 --stop-at-frame 351`) must pass as part of every commit, not just at the end — that's the only way we catch sim/RTL drift while it's still small.

If commit A regresses the boot, revert it immediately and instead try gating BERR on `cpuFC != 3'b111` only — the autovector path is touchy and I want it isolated from the unmapped path.
